# 黑客松回顧報告

**日期:** 2025年11月3日 星期一
**作者:** BMad

### 1. 回顧目標與安全空間

**1. 我們希望透過這次回顧達成什麼？**
*   **總結經驗教訓：** 識別黑客松期間的成功經驗和不足之處，為未來專案提供借鑒。
*   **改進團隊協作：** 發現團隊協作中的優點和痛點，制定改進措施，提升未來合作效率。
*   **解決遺留問題：** 討論並規劃如何處理黑客松期間未能解決的技術或流程問題。
*   **慶祝成就：** 肯定團隊的努力和取得的成果，提升團隊士氣。
*   **規劃未來：** 為「AI 詐騙辨識與預警系統」的後續發展提供方向和建議。

**2. 如何確保每位成員都能自在地分享想法和感受？**
*   **匿名回饋機制：** 若有成員不便公開表達，可提供匿名提交意見的管道。
*   **主持人引導：** 由主持人確保討論氛圍積極，鼓勵每位成員發言，並控制討論方向，避免跑題。
*   **輪流發言：** 確保每個人都有平等的發言機會，避免少數人主導討論。
*   **尊重差異：** 鼓勵不同觀點的表達，並強調理解而非評判。

**3. 我們將秉持哪些原則來進行這次回顧？**
*   **「假設善意」原則：** 相信每位成員在黑客松期間都已盡力做到最好。
*   **「專注於學習」原則：** 回顧的目的是從經驗中學習，而非追究個人責任。
*   **「開放與誠實」原則：** 鼓勵成員坦誠分享真實的想法和感受。
*   **「行動導向」原則：** 討論的最終目標是制定具體的改進行動計畫。
*   **「時間管理」原則：** 嚴格控制每個環節的時間，確保回顧高效進行。

### 2. 收集數據與事件

**1. 「做得好的地方」：哪些方面超出了預期，帶來了積極的結果？**
*   **核心功能快速實現：** 在短短一週內，成功搭建了 AI 詐騙辨識模型並完成了 Web 介面 MVP，超出了最初的預期。
*   **團隊協作效率高：** 團隊成員之間溝通順暢，分工明確，遇到問題能迅速協調解決。
*   **技術棧選擇得當：** Python 生態系統的豐富資源和 Flask/FastAPI 的輕量級特性，加速了開發進程。
*   **問題意識強烈：** 團隊對新竹市詐騙問題有深刻理解，激發了成員的使命感和解決問題的熱情。
*   **演示準備充分：** 簡報內容結構清晰，視覺化效果良好，為最終展示奠定了基礎。

**2. 「可以改進的地方」：哪些方面沒有達到預期，或可以做得更好？**
*   **數據獲取與標註：** 雖然有初步數據，但真實、大量的詐騙數據獲取困難，導致模型訓練數據不足，影響了模型的泛化能力。
*   **模型優化時間不足：** 黑客松時間限制，未能對 AI 模型進行更深入的調優和實驗，誤報率和召回率仍有提升空間。
*   **使用者體驗設計：** MVP 介面功能優先，使用者體驗和互動性較為簡陋，未能充分考慮所有使用者情境。
*   **技術文件完整性：** 由於時間壓力，部分技術細節和部署文檔未能完全完善。
*   **壓力管理：** 部分成員在後期開發中感受到較大壓力，需要更好的壓力緩解機制。

**3. 「遇到的阻礙」：哪些外部或內部因素阻礙了團隊的進展？**
*   **外部阻礙：**
    *   **數據隱私與合法合規：** 獲取真實通訊數據的法律和倫理限制。
    *   **黑客松時間限制：** 一週時間對於開發複雜的 AI 系統來說非常緊迫。
    *   **計算資源有限：** 缺乏高性能 GPU 進行大規模模型訓練和實驗。
*   **內部阻礙：**
    *   **初期技術選型猶豫：** 在某些技術細節上花費了額外時間討論。
    *   **部分成員經驗不足：** 在特定技術領域，部分成員需要更多指導和學習時間。

**4. 「學到的新事物」：個人或團隊在技術、協作或領域知識方面有何收穫？**
*   **技術方面：**
    *   深入了解 NLP 在詐騙辨識中的應用和挑戰。
    *   掌握了 Flask/FastAPI 快速開發 API 的技巧。
    *   熟悉了 Docker 容器化部署的基本流程。
    *   學習了如何快速搭建和迭代 AI 模型。
*   **協作方面：**
    *   學會了如何在時間壓力下高效溝通和協作。
    *   理解了敏捷開發中快速迭代和驗證的重要性。
    *   提升了跨職能團隊合作的能力。
*   **領域知識方面：**
    *   對新竹市的詐騙現況和常見手法有了更深入的認識。
    *   了解了政府開放數據在解決社會問題中的潛力。

### 3. 分析原因與洞察

**1. 對「可以改進的地方」和「遇到的阻礙」進行根本原因分析：**

*   **數據獲取與標註困難：**
    *   **為什麼？** 詐騙數據敏感，涉及用戶隱私，難以公開獲取；人工標註耗時且成本高。
    *   **根本原因：** 缺乏合法、高效的數據獲取與標註管道和工具。
*   **模型優化時間不足：**
    *   **為什麼？** 黑客松時間短，且初期將時間花費在基礎架構搭建和數據處理上。
    *   **根本原因：** 時間管理和資源分配在初期階段未能充分預留模型優化時間。
*   **使用者體驗設計簡陋：**
    *   **為什麼？** 團隊重心放在核心功能實現和技術驗證，對 UI/UX 的投入不足。
    *   **根本原因：** 缺乏專職 UI/UX 設計師，且黑客松目標優先級設定為功能而非體驗。
*   **技術文件完整性不足：**
    *   **為什麼？** 時間壓力大，開發任務繁重，文件撰寫被排在較低優先級。
    *   **根本原因：** 缺乏自動化文件生成工具，且團隊對文件重要性認知不足。
*   **壓力管理：**
    *   **為什麼？** 時間緊、任務重，且部分成員對新技術不熟悉，導致焦慮。
    *   **根本原因：** 黑客松高強度環境，缺乏有效的團隊壓力緩解機制和新技術學習支援。

**2. 分析「做得好的地方」背後的成功因素：**

*   **核心功能快速實現：**
    *   **成功因素：** 團隊成員技術能力互補，Python 生態系統成熟，技術棧選擇輕量高效，目標明確。
*   **團隊協作效率高：**
    *   **成功因素：** 良好的溝通機制（每日站會、即時通訊），明確的分工，成員間相互信任和支持。
*   **技術棧選擇得當：**
    *   **成功因素：** 團隊對主流技術有一定了解，能根據專案需求快速做出合理選型。
*   **問題意識強烈：**
    *   **成功因素：** 專案選題貼近社會熱點，團隊成員對解決社會問題有共同願景。
*   **演示準備充分：**
    *   **成功因素：** 團隊重視成果展示，提前規劃簡報結構，並進行多次演練。

### 4. 制定行動計畫

**1. 針對每個洞察，我們可以採取哪些具體的行動？**

*   **數據獲取與標註困難：**
    *   **行動：** 積極與新竹市政府、警政單位溝通，探討在合法合規前提下獲取匿名化詐騙數據的可能性。
    *   **行動：** 研發或引入半自動化數據標註工具，提升標註效率。
*   **模型優化時間不足：**
    *   **行動：** 在未來專案規劃中，為模型訓練和調優預留更充足的時間。
    *   **行動：** 探索使用雲端 GPU 資源，加速模型實驗。
*   **使用者體驗設計簡陋：**
    *   **行動：** 招募或培訓團隊成員具備基礎 UI/UX 設計能力。
    *   **行動：** 參考優秀的防詐應用，進行介面優化和使用者流程設計。
*   **技術文件完整性不足：**
    *   **行動：** 導入自動化文件生成工具（例如：Sphinx for Python），並將文件撰寫納入開發流程。
    *   **行動：** 制定文件撰寫規範和審閱流程。
*   **壓力管理：**
    *   **行動：** 建立團隊內部知識分享機制，幫助成員學習新技術。
    *   **行動：** 定期組織團隊建設活動，緩解壓力，增強凝聚力。

**2. 這些行動的優先順序是什麼？**
*   **高優先級：** 數據獲取與標註、模型優化時間預留、基礎 UI/UX 優化。
*   **中優先級：** 技術文件完整性、壓力管理機制。

**3. 誰將負責執行這些行動？**
*   **數據獲取與標註：** 專案經理 (PM) 負責對外溝通，數據專家負責工具研發。
*   **模型優化時間預留：** 專案經理 (PM) 和技術主管 (TEA) 共同負責。
*   **使用者體驗設計：** 開發人員 (Dev) 和 UX 設計師（若有）共同負責。
*   **技術文件完整性：** 開發人員 (Dev) 負責撰寫，技術主管 (TEA) 負責審閱。
*   **壓力管理：** 專案經理 (PM) 和團隊所有成員共同負責。

**4. 預計何時完成？**
*   **數據獲取與標註：** 1 個月內完成初步溝通，3 個月內引入或研發工具。
*   **模型優化時間預留：** 下一個專案迭代開始時實施。
*   **使用者體驗設計：** 2 週內完成介面優化方案，1 個月內完成 MVP 介面改版。
*   **技術文件完整性：** 2 個月內導入工具並完成核心文件撰寫。
*   **壓力管理：** 持續進行。

**5. 如何衡量這些行動的成效？**
*   **數據獲取與標註：** 衡量可用的詐騙數據量、標註效率提升百分比。
*   **模型優化時間預留：** 衡量模型訓練和調優時間佔總開發時間的比例是否增加。
*   **使用者體驗設計：** 透過使用者回饋、介面滿意度問卷等方式衡量。
*   **技術文件完整性：** 衡量文件覆蓋率、審閱通過率。
*   **壓力管理：** 透過團隊滿意度調查、成員流失率等指標衡量。
